# 画像アセット計画

README や GitHub Pages の紹介に使う画像は、この `assets/` ディレクトリに配置します。

方針:
- 画像は「かっこよさ」よりも、単一 HTML で動く実用ツールだと一目で伝わることを優先します。
- スクリーンショットは UI の情報量を残しつつ、重要なパネルや数値が読める解像感を保ちます。
- 加工を強くしすぎず、実アプリの見た目と乖離しないことを重視します。

推奨ファイル:

## `landing-hero.png`

- GitHub Pages のランディングページ全体が分かるヒーロー画像。
- ダークな背景、巨大な `PointCloudWorkbench.` タイポ、レーザーグリーンのアクセント、縦の軸線、3つの CTA ボタンがきちんと見える構図にする。
- 画面上部から中段までを広く使い、「単一 HTML の点群ワークベンチ」「すぐ試せる public demo」という印象が一目で伝わることを狙う。
- ボタンは `実アプリを起動`、`SAMPLE LAS`、`REPOSITORY` の役割差が分かる状態にする。
- 単なるページの記録ではなく、「公開トップとしての完成度」が伝わる静止画にする。

## `preflight-panel.png`

- アプリ本体で同梱サンプルを選んだ直後の品質確認画面を撮る。
- ファイル名、ファイルサイズ、元点数、表示率、読込経路、危険度、品質プリセットが同時に見える構図にする。
- 「いきなり読み込むツールではなく、読込前に判断材料が出る」ことが画像だけで理解できる必要がある。
- 重要なのは、危険度表示や quality 選択 UI が読めること。画面全体より、判断パネル中心でよい。
- この画像は README 上で「ただ表示するツールではなく、読込前評価がある」証拠として使う。

## `workspace-3d.png`

- 3D 表示が成立しているメインワークスペースの画像。
- 点群は奥行きと密度が感じられる視点にし、標高色分けか分類色分けのどちらかが見栄えよく出る状態にする。
- 画面にはビューワ領域だけでなく、主要なコントロールや状態表示も少し残し、ブラウザで完結するワークベンチ感を出す。
- 「デモっぽい小さな UI」ではなく、「実用できる観察ツール」に見えることを重視する。
- 可能なら点群の形に高低差や構造感が分かるサンプル状態を選ぶ。

## `workspace-2d-slice-stats.png`

- 2D 表示またはスライス断面表示と、統計パネルや分類関連の情報が同時に伝わる画像。
- 3D だけで終わらず、断面確認や数量確認までできることが一目で分かる構図にする。
- 統計パネルでは、点数、分類内訳、処理時間など、定量情報が読める程度の大きさを確保する。
- 「見るだけ」ではなく「検証できる」ツールだと伝える役割を持たせる。
- README に貼る場合は、3D 画像と対になる存在として機能するのが理想。

補足:
- README に実際の画像を貼る際は、まず上記のファイル名で配置し、その後 Markdown 画像リンクを追加すると差し替えが簡単です。
- 画像が揃うまでは、README にはファイル名と説明だけを残しておけばよいです。
